# Analyzer Contract V3

Version: `3.0`

## Boundary

Analyzer Contract V3 limits the analyzer LLM to independent linguistic observations.

The analyzer may observe:

- primary and secondary intent;
- explicitly stated lead facts;
- requested conversational capabilities;
- requested external actions;
- explicitly requested product;
- objection candidate;
- purchase-readiness signal;
- prior-conversation reference;
- post-purchase signal;
- evidence spans and confidence.

The analyzer must not decide:

- product fit;
- recommended product;
- sales stage;
- macro action;
- micro action;
- CTA;
- next question;
- RAG routing;
- closing permission.

## Banned LLM Fields

The V3 analyzer schema forbids:

- `has_objection`
- `references_prior_message`
- `explicit_turn_number`
- `explicit_start_intent`
- `action_requirement`
- `known_product_fit`
- `recommended_product`
- `sales_stage`
- `macro_action`
- `micro_action`
- `cta_type`
- `next_question`
- `next_question_key`
- `needs_rag`

## Evidence Rules

Evidence spans are mandatory for:

- each requested external action;
- explicit product requests;
- objection candidates when type is not `none`;
- `high` and `explicit_start` purchase readiness;
- prior references when type is not `none`;
- post-purchase signals.

Runtime validation checks that required evidence spans appear in the current user message after basic normalization for case, accents, punctuation and whitespace.

## Shadow Parser

Phase 1 adds a lightweight parser before the analyzer:

```text
load_memory
→ parse_explicit_signals_shadow
→ analyze_turn
```

The parser is non-authoritative:

```text
shadow_parser_may_observe = true
shadow_parser_may_override = false
shadow_parser_may_choose_product = false
shadow_parser_may_choose_action = false
```

Parser candidates are recorded in response metadata for later comparison. They are not fed to the LLM and do not modify planner behavior.

## Runtime Compatibility

The existing runtime still consumes `TurnAnalysis`. The OpenAI analyzer produces `AnalyzerTurnObservation`, then Phase 2 derives `NormalizedTurn` before converting the normalized result into planner-compatible `TurnAnalysis`.

Those compatibility aliases are derived from the normalized turn. They are not independently generated by the LLM.

Phase 3 is responsible for adapting the Sales Policy Planner to consume normalized turn inputs more directly.
